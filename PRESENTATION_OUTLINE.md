# Структура презентации на 8 слайдов

## 1. Научный клубок

Карта знаний R&D для горно-металлургических исследований.

Ключевая фраза: "Мы превращаем разрозненные технические документы в проверяемую карту фактов, источников и пробелов".

## 2. Боль

- Знания рассыпаны по отчетам, статьям, обзорам и архивам.
- Команды повторяют обзоры и эксперименты.
- Сложно найти связь между материалом, процессом, условиями и результатом.
- Нет единого места, где видно подтвержденные выводы, противоречия и экспертов.

## 3. Решение

Единая R&D memory system:

- ingest документов;
- извлечение фактов;
- граф знаний;
- Expert-узлы и геоклассы практики;
- проверяемый поиск;
- аналитический ответ;
- матрица пробелов.
- пополнение базы через review queue.

## 4. Почему это не RAG-чат

| RAG-чат | Научный клубок |
| --- | --- |
| Ищет похожие фрагменты | Строит структурированные факты |
| Может ответить без явной модели знаний | Хранит сущности, связи, условия и результаты |
| Трудно увидеть пробелы | Показывает матрицу покрытия |
| Ответ сложно аудировать | Есть источник, chunk, цитата, confidence |

## 5. Архитектура MVP

Показать цепочку:

`PDF/DOCX -> ingest -> extracted txt -> LLM facts -> graph -> query/analytics -> Streamlit UI`

Подчеркнуть, что это рабочий вертикальный срез, а не только концепт.

## 6. Демо и метрики

- 25 документов в демо.
- 152 txt-документа подготовлены ingest-слоем.
- 5226 узлов.
- 21980 связей.
- 531 Expert-узел.
- 2524 result-факта.
- 2195 фактов с условиями.
- 970 фактов с географией.
- Практика: domestic 414, foreign 344, mixed 28, unknown 1738.

Скриншоты: обзор, карточка факта, статус ТЗ-пресета, аналитический ответ.

## 7. Матрица пробелов

Главный слайд отличия:

- система показывает, что известно;
- система показывает, чего не хватает;
- руководитель видит зоны риска и темы для новых исследований;
- исследователь видит, где нужны источники или экспертиза.

## 8. Пополнение, Качество и Roadmap

- Эксперт загружает PDF/DOCX/TXT или вводит факт вручную.
- Новые данные попадают в `expert_submissions.jsonl` со статусом needs_review.
- Approve/reject есть в UI; основной граф не меняется без отдельной процедуры merge.
- Мини-валидация: 20+ фактов со статусом верно / частично / ошибка.
- Текущий статус: auto_extracted, не экспертно утвержденная база.
- NetworkX -> Neo4j/JanusGraph.
- Auto-extracted -> expert verified.
- Quarantine -> approved graph merge с аудитом.
- Базовая география -> справочник стран, площадок и domestic/foreign/mixed/unknown.
- Частичная обработка чисел -> нормализация единиц.
- MVP UI -> роли, аудит, ACL, ручная правка графа.

Финальная фраза:
"Мы доказали ядро платформы: проверяемое извлечение фактов, граф, поиск, аналитический ответ и пробелы. Production-слой требует графовой БД, экспертной верификации, ACL, аудита и нормализации единиц".
